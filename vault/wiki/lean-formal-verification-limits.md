# Lean形式検証の限界 — 「誰が番人を見張るのか」

Lean（定理証明支援系）で検証済みとされたzlib実装に、AIエージェント（Claude）が105百万回以上のファジングでバグを2件発見した事例から学ぶ、形式検証の本質的な限界。

## 発見されたバグ

### バグ1：Lean実行時のヒープバッファオーバーフロー

```c
// lean_alloc_sarray関数の脆弱性
// バッファサイズがSIZE_MAXに近い値で整数オーバーフローが発生
size_t alloc = base_size + elem_size * n;  // ラップアラウンドの危険
```

全Lean 4バージョンに影響。修正PR進行中（2026年4月時点）。

### バグ2：アーカイブパーサーのDoS

ZIP中央ディレクトリの`compressedSize`フィールドが検証なしに使用され、メモリ不足パニックを引き起こす。

## 形式検証が守れなかった理由

| 問題 | 内容 |
|------|------|
| 検証スコープ外 | 圧縮・展開パイプラインは検証されたが、アーカイブパーサーは対象外 |
| 信頼計算基盤 | Lean証明はLean自体のC++ランタイムが正しいことを前提とする |
| 境界の問題 | 「何を検証するか」の選択が、何を見落とすかを決定する |

## 根本的な洞察

> 形式検証は「**証明した命題の範囲内では絶対に正しい**」ことを保証する。しかし「**どの命題を証明すべきか**」は人間が決める。

「誰が番人を見張るのか（Quis custodiet ipsos custodes?）」というユウェナリスの問いと同様、検証システム自体の正当性を問い直す必要がある。

## 実践的な意味

- **形式検証とテストは補完関係**：検証がある部分を「テスト不要」とするのは誤り
- **AIによるファジングの有効性**：Claude + 大量反復により、人間が見落とした境界条件を発見できる
- **信頼境界の明示化**：どこまでを「信頼する」かを明示的に文書化することが重要
