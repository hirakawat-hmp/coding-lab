# エージェンティック・コーディングは罠か

AIに大量のコードを書かせる「Agentic Coding」のアプローチに潜む構造的問題を論じた概念。Lars Faye（2026年5月）による警告。

## 中核的矛盾

AIが自動生成したコードを監督するためには、そのコードを自分で書けるスキルが必要になる。AIへの依存が深まるほど、監督に必要な能力が失われていくという逆説。

## 確認されているリスク

| リスク | 内容 |
|--------|------|
| スキル衰退 | AIツール使用で数ヶ月以内にデバッグスキルが47%低下（著者調査） |
| ベンダーロックイン | トークンコストへの依存が継続的な経費圧力に発展 |
| 学習機会の喪失 | ジュニア開発者が実装を通じた学習機会を逃す |
| 品質保証の困難 | 自分でレビューできない量のコードが生成される |

## "ship computer" 比喩

著者が提唱する代替哲学：AIを「ナビゲーションコンピュータ」として使い、「船を動かす」のは自分。計画と仕様生成にAIを活用しつつ、実装は手動で行う。「レビュー可能な範囲のコードのみ生成すること」が核心。

## 代替アプローチ

1. **計画フェーズ**: AIを活用（仕様作成・設計レビュー）
2. **実装フェーズ**: 自分で書くか、1関数・1ファイル程度の小単位のみAIに依頼
3. **常に関与**: 実装プロセスへの継続的な参加を維持

## 反論と限界

- 熟練開発者にとってはスキル維持が問題にならない可能性
- ルーティンコード（ボイラープレート）への適用はリスクが低い
- チームによっては経験者のレビュープロセスが代替になりうる
