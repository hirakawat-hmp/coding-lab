# TriAttention

LLM のロングコンテキスト推論を高速化する **KV キャッシュ圧縮手法**。通常の注意機構が RoPE 適用後の Q/K ドット積で重要トークンを判定するのに対し、TriAttention は **RoPE 適用前の空間で Q/K ベクトルが固定中心に集中する**という性質を利用した三角関数スコアリングを使う。vLLM と統合済みで、インストール後に環境変数で自動有効化できる。

## KV キャッシュ圧縮の背景

ロングコンテキスト LLM では、シーケンス長に比例して KV キャッシュのメモリ使用量と Attention 計算コストが増加する。不要なトークンの KV を削除（圧縮）することで推論を効率化できるが、どのトークンが「重要か」を正確に判定することが難しい。

### 従来手法の問題

RoPE（Rotary Position Embedding）は位置情報をエンコードするために Q/K ベクトルを連続的に回転させる。この回転のせいで **ポスト-RoPE の注意パターンは不安定**であり、どのトークンが重要かを事前予測しにくい。

## TriAttention の仕組み

### プリ-RoPE における Q/K ベクトルの集中性

多くの Attention ヘッドで、**RoPE 適用前**の Q/K ベクトルはゼロでない固定中心周辺に高度に集中している（位置によらず安定）。この集中性を利用すると、実際に Attention を計算せずにトークンの重要度を推定できる。

### 三角関数スコアリング

Q/K が中心周辺に集中する状況では、Attention ロジットは **Q と K の距離のみに依存する関数** に近似できる。この距離依存性を三角関数級数でモデル化することでスコアを導出する。

### デュアルコンポーネントスコア

最終的な重要度スコアは 2 要素の組み合わせ:

| コンポーネント | 捉えるもの |
|-------------|----------|
| **距離ベーススコア**（三角関数） | どの距離のトークンが注目されるか |
| **ノルムベーススコア** | トークンの際立ち（Q/K ベクトルの大きさ） |

集中度メトリクスが高い（集中が強い）ヘッドでは三角関数スコアを優先し、そうでないヘッドではノルムスコアを重視する自動重み付け。

## パフォーマンス

### Qwen3-8B でのスループット

| ベンチマーク | TriAttention | フルアテンション | 高速化率 |
|------------|------------|--------------|--------|
| MATH-500 | 1405.2 tok/s | 222.8 tok/s | **6.3×** |
| AIME25 | 40.8% 精度 | 40.8% 精度 | **2.5×** |
| AIME24 | - | - | **1.9×** |

### メモリ削減

- KV キャッシュメモリ: **10.7× 削減**（フルアテンション比）
- MATH-500（32K 中 1,024 トークン保持）: 68.4% vs フルアテンション 69.6%（−1.2 ポイントのみ）

### 他の KV 圧縮手法との比較

| 手法 | AIME25 精度 |
|-----|----------|
| **TriAttention** | **40.8%** |
| R-KV | 17.5%（半分以下） |
| SnapKV | 低い |

## vLLM での利用方法

```bash
pip install triattention
export ENABLE_TRIATTENTION=1
```

インストール後に環境変数を設定するだけで既存の vLLM ワークフローに適用可能。API 変更は不要。
