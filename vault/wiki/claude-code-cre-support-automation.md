# Claude Code で CRE 非技術者がサポート調査を完全自動化した実録

## 概要

TOKIUM の CRE（Customer Reliability Engineering）チームに在籍する非技術系メンバーが、DB クエリ・コード解析・ログ調査が必要なカスタマーサポート業務を Claude Code で自動化した事例（Zenn、2026年4月29日）。

## 変化の軌跡

| 段階 | 内容 | 調査時間 |
|------|------|----------|
| 初期 | 手動調査 | 1〜2時間/チケット |
| Skills 整備後 | Claude Code 補助 | 15〜30分/チケット |
| 完全自動化後 | 2つのロボットが自律実行 | ほぼゼロ（人間は確認のみ） |

## Skills パターンの活用

繰り返しの調査手順を「Skills」として文書化。初期 3個 → 現在 40個超。

**[[wiki/skill-md-pattern]]** で定義されるパターンを実務で徹底活用した事例。

## 完全自動化の構成

- **調査ロボット**: 毎時チケット調査を実行
- **学習ロボット**: 15分ごとにフィードバックを処理
- 信頼度評価（★〜★★★★★）でアウトプット品質を可視化
- MCP で Jira、Datadog、Slack、Notion を統合

## 失敗から学んだこと

AI は自信を持って**誤った答えを返す**。具体的な失敗:
- 存在しないクラス名の提示
- データが存在しないという誤った主張
- 不完全な抽出条件の見落とし

**対策**: 厳格な自己検証ルール（コード定義の直接確認、DB クエリで非存在を証明、未検証の主張には明示的マーク）

## AI が代替できなかった3領域

1. **責任**: 間違えた時に誰かが責任を取る必要がある
2. **根本動機**: 目的関数そのものを生み出すのは人の欲望
3. **関係構築**: 信頼・感情・交渉は人間の領域

## 核心的洞察

> 「AI は与えられた目的関数を最適化できるが、目的関数そのものを生み出すのは人の欲望だ」
