# テキストモードの嘘：モダンTUIはアクセシビリティの悪夢

「テキストベースのツールはアクセシブルだ」という誤解を解く。モダンなTUIフレームワーク（特にReact Ink系）は視覚障害者にとってCLIより使いにくい場合がある。

## CLI と TUI の本質的な違い

| 種別 | 構造 | スクリーンリーダーとの相性 |
|------|------|--------------------------|
| CLI | ストリーム（線形） | ◎ 良い |
| TUI | 2次元グリッド（キャンバス） | △ 問題が多い |

## なぜモダンTUIは問題か

React Inkのようなフレームワークは画面を**リアクティブキャンバス**として扱う：

- 更新のたびに画面全体を再描画
- スピナー・タイマーの頻繁な更新が止まらない
- スクリーンリーダーが「更新があった」と読み上げ続ける
- 結果として聴覚ユーザーが情報を受け取れない

## アクセシブルなTUIの特性

成功例（nano・vim・menuconfig・Irssi）の共通点：

1. **カーソル非表示オプション**があること
2. **単一列フォーカス**（2次元を意識しなくていい）
3. VT100スクロール領域などのハードウェア機能を活用
4. 更新頻度を最小化し、変化を明示的に発火

## 開発者への示唆

- モダンなTUIフレームワークを使うときはアクセシビリティモードの実装を検討
- スクリーンリーダーテストは開発初期から行う
- 「テキストベース = アクセシブル」という思い込みを捨てる
