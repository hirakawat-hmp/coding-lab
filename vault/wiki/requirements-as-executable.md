# 要件を動くものにする（Requirements as Executable）

AI時代のソフトウェア開発では、要件はドキュメントだけでなくコード・型・テスト・スキーマ・CI/CDに「動くもの」として分散して存在すべき、という設計原則。

## 核心的洞察

「要件が死ぬのは書かれないからではなく、書かれても読まれないから」

AIエージェントは暗黙の文脈を推論できないため、曖昧な要件が即座に壊れた実装として可視化される。これは逆に、要件の曖昧さを検出するセンサーとして活用できる。

## 5つのレイヤーへの分散

| レイヤー | 場所 | 検証タイミング |
|---------|------|-------------|
| プロジェクト | `CLAUDE.md`（ミッション・制約） | 常時 |
| ドメイン | ドメイン文書・スキル | AI実行時 |
| フィーチャー | コミットメッセージ・PRテンプレート | PR作成時 |
| 手順 | CIワークフロー・lintルール | CI実行時 |
| 振る舞い | 実行可能テスト（E2E・統合・プロパティベース） | テスト実行時 |

## 2種類のシステムの組み合わせ

- **決定論的強制:** 型・テスト・lint・コントラクト → 違反を防ぐ
- **確率論的ガイダンス:** プロンプト・スキル・ガイドライン → 良い解決策に誘導

この組み合わせで人間の認知負荷を最小化。

## 非機能要件の6W フレームワーク

曖昧な要件を具体化する:
**When（いつ）· Who（誰が）· Where（どこで）· What（何を）· How（どのように）· How Much（どれだけ）**

例: 「攻撃者がプロダクションAPIに繰り返し認証試行した際（When）、システムはX時間以内にアカウントをロックして管理者に通知する（How Much）」

## AI時代の品質次元（新登場）

- **説明可能性:** モデルの判断を人間が正当化できるか
- **公平性・非差別性:** 出力が偏っていないか
- **規制準拠:** 出力が法的要件を満たすか
