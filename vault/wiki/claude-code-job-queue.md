# Claude Code マルチセッション ジョブキュー（tq）

複数のClaude Codeセッションをジョブキューの概念で管理するオーケストレーションパターン。Hirotaka Miyagiが開発した`tq`ツールの設計思想から学ぶ。

## 問題設定

Claude Codeを複数セッション並列実行する際の課題：
- セッション間の作業調整が手動で煩雑
- コンテキストスイッチが多く人間の集中力が分散
- タスクの優先度・依存関係の管理が難しい
- Git Worktreeの管理が複雑

## tqのアーキテクチャ

### 設計原則

```
人間 → 自然言語でタスク投入
   ↓
tq（ジョブキュー）
   ├── マネージャーエージェント（Claude Code）← タスク分解・調整
   ├── ワーカーセッション1（対話型）
   ├── ワーカーセッション2（対話型）
   └── バックグラウンドセッション群（非対話型、無制限）
```

### コンテキストスイッチの最小化

対話セッション同時起動数をデフォルト**3セッション**に制限。人間が監視・承認できる上限が3という経験則。非対話セッション（自動実行）はバックグラウンドで無制限実行可能。

### エージェント中心のCLI設計

```json
// 全出力がJSON形式
{"status": "queued", "job_id": "abc123", "task": "fix auth bug"}
```

人間の直接操作ではなく、マネージャーエージェントが読むためのインターフェース設計。

## 実践的な活用パターン

### 割り込み処理

作業中に気づいた別タスクをキューに積む（コンテキストスイッチなし）：
```bash
/tq:create-action "fix typo in README while working on feature"
```

### GitHub自動処理

5分ごとにNotificationをチェックし、コンフリクト修正やCI復旧を自動実行するループ。

### Ralph Loop（反復実行）

定型作業をセッション分割しながら繰り返す。コンテキストウィンドウの圧迫を防ぎ、長時間処理を安定実行できる。

## 既存ツールとの比較

| ツール | カテゴリ | 特徴 |
|-------|---------|------|
| claude-squad | セッションマネージャー | 人間がUIで管理 |
| TAKT | ワークフロー定義 | 事前定義されたDAG |
| **tq** | **ジョブキュー** | **自然言語指示による動的管理** |

tqは中間に位置し、自然言語指示による柔軟な並列タスク管理を実現する。

## 設計から学べること

1. **「薄いオーケストレーション層」の価値**：ジョブキュー機能だけ提供し、Claude Code固有機能はそのまま活用する
2. **人間の認知限界を考慮したデフォルト値**：並列数3という制限が重要
3. **エージェントが読みやすいインターフェース設計**：JSON出力専一化
