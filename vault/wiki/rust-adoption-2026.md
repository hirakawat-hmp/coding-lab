# Rust 採用の現実：C++ からの移行で見えてきたこと（2026年）

Voluntas による実際の Rust 採用レポート（2026年5月）。C++ から Rust への移行を決断した企業の視点から、採用理由・メリット・デメリットを率直に分析。

## 採用を決めた3つの理由

### 1. 社内エキスパートの存在
Rust 経験10年以上、分散システムを構築できる社員が在籍していた。
→ **教訓**: 新言語採用の最大のリスクは「誰も深く知らない」こと。エキスパートがいる環境は有利。

### 2. C++ ビルドシステムへの疲弊
C++ のビルドツールを Python で書き続けることへの疲労感。Cargo の標準化・一貫性は魅力的。
→ **教訓**: 言語そのものより、ツールチェーンの品質が長期的な開発体験を左右する。

### 3. LLM による Rust サポートの向上
「Rust はコンパイラが厳しくて難しい」という懸念とは反対に、LLM（Claude 等）がコンパイルエラーを効果的に解決してくれる。
→ **教訓**: Rust のコンパイラエラーメッセージの明確さは、LLM による補助と非常に相性が良い。

## 実際に良かった点

| ツール/ライブラリ | 用途 |
|-----------------|------|
| build.rs | ビルドカスタマイズ |
| fmt + clippy | コード品質の自動チェック |
| Tokio | 非同期ランタイム |
| Rustls | TLS サポート |
| proptest + fuzzing | テスト |
| tracing | ログ・観測性 |

## 実際の欠点

1. **target ディレクトリの肥大化**: ディスク消費が過剰（.gitignore 要確認）
2. **依存関係の急速な増大**: crates.io の依存チェーンが深くなりがち
3. **GitHub Actions キャッシュが効きにくい**: ビルドキャッシュの最適化が必要

## 2026年の Rust 採用環境

- Microsoft、Google、Linux カーネル等がメモリ安全性の観点で Rust 採用を加速
- LLM ツールの充実により、Rust の学習曲線が緩和されている
- [[wiki/cpp26]] — C++26 もリフレクション・メモリ安全性を強化しているが、Rust との差は縮まりつつも残存
