# Tailscaleの技術的問題集：やめたくてもやめられない

Tailscaleを使い続けながら蓄積された技術的批評。DNS・ファイアウォール・プロトコル実装の7つの問題と、それでも乗り換えを躊躇する理由。

## 7つの技術的問題

### 1. DNS解決の問題
- systemd-resolvedとの競合でDNS失敗
- 検出ロジックが脆弱、複数リゾルバー設定と非互換

### 2. RFC非準拠
- DNSプロキシがEDNS標準を無視
- 過大サイズのレスポンス返す
- スプリットブレインキャッシュ → ツール間で不整合な結果

### 3. CGNATレンジのハードコードブロック
- 100.64.0.0/10 のすべてのトラフィックをドロップ（ユーザー設定無視）
- このレンジを内部で正当に使っているシステムがブリックする

### 4. 攻撃的ファイアウォール操作
- iptablesルールを定期的にチェーンの先頭に再配置
- 他ファイアウォール管理ツール（Calico等）と競合

### 5. 固定MTU 1280
- ユーザー設定不可
- 他プロトコルとの組み合わせで接続問題
- TCP MSS clampingで回避が必要

### 6. 限定的ACL機能
- 明示的な拒否ルールが書けない
- 軽微な変更に全ポリシー書き直しが必要

### 7. 強制SSO
- 外部IdP（アイデンティティプロバイダー）経由が必須
- 組織移行時に複雑化

## 代替案の現実

- **Headscale**（セルフホスト）: 機能・メンテナンスコストが高い
- **WireGuard手動設定**: 初期・継続コストが高い

## 結論

「明日もまたTailscaleを使う。文句を言いながら使う方が手動管理より簡単だから」

→ 問題が多くても乗り換えコストがさらに高い「ベンダーロックイン」の実例。
