# AI 時代のエンジニアリングシフト：実装から「届け方」へ

AI が実装コストを劇的に下げる時代において、エンジニアの価値は「どう作るか」から「どう届けるか」に移行しつつある。

## 核心的な気づき

> 「スタート（課題）とゴール（リリース形態）がわかれば顧客と話せる」

複数のリリース形態を理解することで、顧客との対話の質が根本的に変わる。

## リリース形態の多様性

エンジニアが知るべきデプロイ先の選択肢:

| 形態 | 用途 |
|------|------|
| Web / Notion | 社内ドキュメント・ナレッジベース |
| Chrome 拡張 / Slack Bot | 生産性向上ツール |
| API / Google Apps Script | 外部統合・自動化 |
| モバイルアプリ | エンドユーザー向けサービス |
| Vercel / Firebase | Webアプリ高速デプロイ |

## エンジニア役割の進化

| フェーズ | フォーカス |
|---------|----------|
| プロンプトエンジニア | 高品質な指示文の作成 |
| コンテキストエンジニアリング | 会話フロー設計 |
| ハーネスエンジニアリング | AI + 外部システムのオーケストレーション |

→ [[wiki/claude-code-harness-patterns]] も参照。

## インフラの民主化

以前はインフラ構築に数週間かかっていた作業が:

- **IaC / Docker**: セットアップ時間を大幅短縮
- **BaaS** (Firebase, Supabase, AWS Amplify): バックエンドの抽象化
- **Vercel**: フロントエンドの即時デプロイ
- **Google Apps Script**: 軽量自動化

## 具体的な成果

メール受信・AI 活用マッチングサービス:
- 従来: 数ヶ月かかる開発
- AI 活用: **ゼロから本番まで約4日**

## 教訓

1. AI は実装の差別化を失わせる → **設計・デリバリー判断**が差別化になる
2. リリース形態の知識がクライアントとの対話力を高める
3. 「作れる」より「いつ何でどう届けるか」を考える習慣が重要
