# Platform Engineering — SREを「努力」から「仕組み」へ

SREチームが個別対応（手作業・チケット対応）から脱却し、開発者が自律的にインフラを操作できる「プラットフォーム」を構築・運用する考え方。

## SREの民主化が失敗するパターン

「SREの民主化」を掲げても、以下の罠にはまると形骸化する：

1. **認知負荷が高すぎる**：新サービスのデプロイにTerraform、CI/CD、監視、セキュリティなど4〜5箇所の変更が必要
2. **依頼文化が定着**：複雑すぎて開発者が自己解決できず、SREへの依頼が常態化
3. **ベストプラクティスの属人化**：各チームが独自の方法を作り、知識が分散

## Platform Engineeringの実装パターン

### Golden Path

標準化された立ち上げパス（テンプレート）を提供。開発者はこのパスに従えば適切なセキュリティ・監視設定が自動的に付いてくる。

### セルフサービス

チケット依頼なしに開発者自身が環境構築できる。`values.yaml`の記述だけで新サービスをデプロイできるのが目標。

### EKS + Helm + GitOps 実装例（ログラス）

```
開発者 → values.yaml を編集
   ↓
Helm Chart（SREが管理する標準テンプレート）
   ↓
ArgoCD/Flux（GitOps）で自動デプロイ
   ↓
セキュリティ・監視・ロギングが自動設定済みの環境
```

**成果**：軽量アプリケーションが10分程度でデプロイ可能に。

### ガードレール

セキュリティポリシーをプラットフォームに組み込み、開発者が意識しなくても守られる設計にする。ルールで縛るより、正しい道が最も簡単な道になるように設計する。

## SREチームの役割変化

| 従来 | Platform Engineering後 |
|------|------------------------|
| インフラ個別対応 | プラットフォーム構築・運用 |
| チケット受付 | セルフサービス仕組み化 |
| インフラに閉じた視野 | システム全体を見る視野 |
| 属人知 | 仕組みとして共有資産化 |

## 本質的な変化

技術的なツール導入より、**チーム間の役割と協業の再定義**が核。権限境界の明確化と、属人知の共有資産化が最重要。
