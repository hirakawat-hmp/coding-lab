# Daily Knowledge Digest — 2026-04-10

## 要約

Claude Code GitHub Actions の設計パターンとベストプラクティスについて調査した。特に v1 での自動モード検出、Tag Mode と Agent Mode の違い、ツール権限管理（allowedTools）、および WebSearch/WebFetch の有効化方法を詳しくまとめた。

## トピック

### 1. Claude Code GitHub Actions v1 の概要

`anthropics/claude-code-action@v1` は Anthropic の Claude Code CLI を GitHub Actions ランナーで実行する公式 Action。2025年9月29日に Claude Code 2.0 と共にリリースされ、Agent SDK をベースに構築されている。

旧 beta 版からの大きな変更点は **モードの自動検出**。`mode: "tag"` や `mode: "agent"` の手動設定が廃止され、ワークフローの設定内容から自動的に適切なモードが選択される。また `direct_prompt` は `prompt` に、各種オプション（`max_turns`、`model` など）は `claude_args` に統合された。

**関連ページ:** [[wiki/claude-code-github-actions]]
**ソース:** [Claude Code GitHub Actions ドキュメント](https://code.claude.com/docs/en/github-actions)

### 2. Tag Mode と Agent Mode の違い

v1 では2つの実行モードが自動検出される:

**Tag Mode（インタラクティブモード）**:
- `@claude` メンションや Issue アサイン、ラベル付与でトリガー
- 進捗トラッキングコメントをリアルタイム更新
- PR の diff・コメント履歴・CI ステータスを自動収集
- `claude/` プレフィックスのブランチを自動作成
- 豊富なデフォルトツールセット（Git 操作、CI 連携など）

**Agent Mode（自動化モード）**:
- ワークフロー YAML の `prompt` 入力で起動
- サイレント実行（GitHub Step Summary のみ出力）
- 現在のブランチで動作（ブランチ自動作成なし）
- 最小権限：`--allowedTools` で明示的に許可したツールのみ

モード検出の優先順位: `track_progress: true` → `prompt` あり → トリガーフレーズあり → 実行なし

**関連ページ:** [[wiki/claude-code-action-modes]]
**ソース:** [Execution Modes - DeepWiki](https://deepwiki.com/anthropics/claude-code-action/3.1-execution-modes)

### 3. allowedTools の設定方法と WebSearch/WebFetch の有効化

ツールアクセスは2層のセキュリティモデルで制御される:

**ツール命名規則**:
- 組み込みツール: `Read`, `Edit`, `Glob` など
- Bash コマンド: `Bash(npm run test:*)` のようにコマンドを指定可能
- MCP ツール: `mcp__github__create_issue` の形式

**WebSearch/WebFetch はデフォルト無効**。有効化には2つの方法がある:

方法1 — `claude_args` での明示的許可:
```yaml
claude_args: |
  --allowedTools "Bash,Edit,Read,Write,WebFetch,WebSearch"
```

方法2 — `settings` JSON の permissions:
```yaml
settings: |
  {
    "permissions": {
      "allow": ["WebSearch", "WebFetch"]
    }
  }
```

**既知の問題（修正済み）**: v1 初期バージョンで `--allowedTools` に WebSearch/WebFetch を指定しても無効化され続けるバグがあった（Issue #690）。PR #1033 で修正済み。

**関連ページ:** [[wiki/claude-code-action-tools]]
**ソース:** [Tool and Permission Configuration - DeepWiki](https://deepwiki.com/anthropics/claude-code-action/5.3-tool-and-permission-configuration)

### 4. Claude Code GitHub Actions の auto-merge 設計パターン

Claude Code Action はデフォルトでは PR をマージしない。自動マージには GitHub リポジトリ設定（`Allow auto-merge`）とブランチ保護ルールの設定、そして `gh pr merge --auto` コマンドの明示的な呼び出しが必要。

主な設計パターン:

**パターン1: GitHub ネイティブ auto-merge**  
`gh pr merge --auto --squash` で PR にフラグを立て、全必須 CI チェック通過後に自動マージする。安全性が高く推奨。

**パターン2: CI 通過後の直接マージ**  
`gh pr checks --watch` で CI 完了を待機してから即座に `gh pr merge` を実行する。

**パターン3: Continuous Claude（自律ループ）**  
ブランチ作成 → コード変更 → PR 作成 → CI 待機 → マージを繰り返す完全自律型。共有ノートファイルでコンテキストを維持し長期タスクに対応。

**無限ループ防止の必須ガード**:  
```yaml
if: !startsWith(github.event.workflow_run.head_branch, 'claude-auto-fix-ci-')
```
Claude の修正ブランチに固定プレフィックスを付けて、それらの CI 失敗が再トリガーしないようにする。

**関連ページ:** [[wiki/claude-code-auto-merge]]  
**ソース:** [Claude Code Auto-Fix: The PR That Fixes Itself](https://paddo.dev/blog/claude-code-auto-fix-pr-lifecycle/)
