### 1. Claude Code PreCompact Hook — auto-compact 問題を根本解決

Claude Code で長時間の議論をしていると、コンテキストウィンドウが逼迫した際に自動圧縮（auto-compact）が走り、「未解決の論点」が「解決済み」として扱われる問題が報告されていた。2026年に追加された **PreCompact Hook** により、圧縮処理への介入が可能になった。`exit 2` で圧縮を差し止め、`decision:replace` でカスタム要約に置換できる。未解決マーカー（`TODO:`, `未決:` 等）を含む行を検出して圧縮を止める bash スクリプトの実装例が共有されている。合わせて Discord 等からの外部割り込み機能「Channels」との組み合わせにより、長時間 agentic タスクの遠隔監視・制御が可能になる。

**関連ページ:** [[wiki/claude-code-precompact-hook]]
**ソース:** [Claude Code が途中で議論を忘れる問題 — auto-compact を PreCompact Hook で制御し、Discord から遠隔操作する](https://qiita.com/nogataka/items/95efda0c7c9ea2405139)

### 2. Codex Automations × 公式プラグイン 90 個で「AI 秘書」を構築

OpenAI の **Codex Automations** は cron 構文で定時にエージェントタスクを実行する機能。Thread Automation（既存スレッド継続）と Standalone Automation（新規タスク）の 2 モードがあり、GitHub・Slack・Notion 等 90 種以上の公式プラグインと連携する。実践者は月 $50 程度で朝 Issue トリアージ・昼 Sentry 調査・夕方 Slack 要約の 3 本を運用し、月 3 万円相当の作業を自動化。落とし穴として OAuth トークン期限切れ・タイムゾーン指定忘れ・曖昧なプロンプトが挙げられており、特に「Asia/Tokyo」明示が重要。

### 3. LLM アプリ設計の「CAP 定理」— 一貫性・可用性・価格は同時に成立しない

分散システムの CAP 定理を LLM アプリ設計に転用したフレームワーク。**C（一貫性）= 同一モデルで品質安定**、**A（可用性）= クォータを超えず継続応答**、**P（価格）= 安価**の 3 要素を同時に満たせないと主張する。CA（Reserved Tier でコスト増）・CP（従量課金で 429 エラーリスク）・AP（複数モデルフォールバックで品質ばらつき）の 3 戦略パターンで設計判断を整理できる。LLM の SLI がスループット型（TPM/RPM）であるため、「お金を積めばスケールアウトで解決」が通用しないことも指摘される。

**関連ページ:** [[wiki/llm-cap-theorem-design]]
**ソース:** [LLMアプリケーションにおける設計Principal - LLMのCAP定理](https://zenn.dev/hsaki/articles/llm-cap-theorem)

### 4. LINE を AI エージェントの入口にする実装パターン

月間 9,800 万 MAU の LINE を AI エージェント UI として活用する。**Messaging API**（REST・Webhook・2 秒制約あり）、**LIFF**（LINE 内 WebView・ストリーミング対応）、**LINE Bot MCP Server**（ローカル AI からの送信）の 3 インターフェースを比較した結果、「基本は Messaging API で十分、Markdown 必要時のみ LIFF へ遷移」が最適パターンとされた。Next.js + @line/bot-sdk + @anthropic-ai/sdk を Vercel にデプロイする構成で実証済み。

### 6. TypeScript 7.0 Beta — Go 言語移植で 10 倍高速化（Publickey 報道）

2026年4月21日に公開された TypeScript 7.0 Beta について Publickey が詳報。コンパイラを TypeScript（自己ホスト）から **Go 言語** で全面再実装し、VS Code（1.5M 行）で 89 秒 → 8.74 秒（10.2 倍）、Sentry で 133 秒 → 16 秒（8.3 倍）を達成。新デフォルトとして `strict: true`・`module: esnext`・`types: []` が設定され、RC は数週間以内、Stable は 2026年6月頃の予定。

### 7. Matz が AI を使って数週間で作った Ruby AOT コンパイラ「Spinel」

RubyKaigi 2026 で発表。Prism で AST を生成し C コードを経由してネイティブバイナリを出力する Ruby AOT コンパイラ。自己ホスト可能で、計算集約的な処理では CRuby より大幅に高速。`eval`・`send`・動的メタプログラミングは未対応で CLI ツール・CI ヘルパー向け。Matz が生成 AI を道具として使い数週間で実装した点が注目されており、「強い設計者 × 生成 AI = 低レイヤー探索の加速」を示す実証事例となっている。

### 8. コードを1行も読ませずに AI に脆弱性を特定させる「Deep Structure Map」手法

生のコードを AI に渡す代わりに、Python `ast` モジュールで抽象構文木を生成し Mermaid + JSON 形式の「Deep Structure Map」に変換して AI に入力する手法。脆弱性は **Source**（外部入力）→ **Propagator**（サニタイズなしの伝播）→ **Sink**（`eval` 等の危険関数）の 3 要素がつながるパスとして定義し、AI にグラフの矛盾探索として解かせる。AI を「コード読解」から「グラフ探索」に特化させることで精度が向上するという主張。
