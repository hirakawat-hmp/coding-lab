### 4. Mayo Clinic REDMOD — AI がすい臓がんを3年前に検出

![Mayo Clinic AI がすい臓がんを検出](https://i.gzn.jp/img/2026/05/06/ai-cancer/00_m.jpg)

Mayo Clinic が開発した AI モデル「REDMOD」が、標準 CT スキャンを解析して臨床診断の最大3年前にすい臓がんを検出。診断16カ月前の段階で73%の検出率を達成し、専門放射線科医の約2倍の成績。2,000件以上の多機関データセットで検証済み。現在は後ろ向き研究のみで、臨床導入には前向き研究が必要。5年生存率11%という難治性がんの解決に向けた重要な一歩。

### 5. バイブコーディングとアジェンティック・エンジニアリングの境界消滅（Simon Willison）

Simon Willison が自身の経験から観察：かつて明確だった「バイブコーディング（コードを理解せずに書く）」と「アジェンティック・エンジニアリング（AI を使いながら品質を維持するプロ開発者）」の区別が実践上で曖昧になっている。プロでも Claude Code が生成した全行をレビューしなくなっている。生産性が200→2,000行/日に向上したことで設計ボトルネックが露わに。開発ライフサイクル全体の再設計が必要とも。

### 6. Google Cloud Fraud Defense — reCAPTCHA の次世代進化（エージェント型 Web 対応）

![Google Cloud Fraud Defense](https://storage.googleapis.com/gweb-cloudblog-publish/images/GCN26_102_BlogHeader_2436x1200_Opt_1_Dark.max-2500x2500.jpg)

Google Cloud が reCAPTCHA の後継「Google Cloud Fraud Defense」を発表。AI エージェントが自律的にトランザクションを実行する時代に対応し、正規エージェントは許可しつつ悪意ある自動化をブロック。QR コードベースの新チャレンジ、エージェントポリシーエンジン、アカウント乗っ取り51%削減を実現。既存 reCAPTCHA ユーザーは移行不要・価格変更なし。

**関連ページ:** [[wiki/google-cloud-fraud-defense]]
**ソース:** [Introducing Google Cloud Fraud Defense](https://cloud.google.com/blog/products/identity-security/introducing-google-cloud-fraud-defense-the-next-evolution-of-recaptcha/)

### 7. Val Town の認証遍歴：Supabase → Clerk → Better Auth

Val Town（JavaScript サンドボックスサービス）が3度目の認証システムに移行完了。Clerk から Better Auth へ。Clerk の問題：①ユーザーテーブルを持つソーシャル機能との根本的不一致、②アカウント全体で毎秒5リクエストの厳しい制限、③Clerk ダウン時に全ログインユーザーがアクセス不能。Better Auth はローカルでセッション管理し単一障害点を排除。「信頼性はその重要パーツの信頼性の最小値」という教訓。

**関連ページ:** [[wiki/val-town-better-auth-migration]]
**ソース:** [From Supabase to Clerk to Better Auth](https://blog.val.town/better-auth)

### 8. Microsoft Edge が全パスワードを平文 RAM 展開——意図的設計と確認

ノルウェーの研究者が発見：Edge は起動時に保存されているすべてのパスワードを平文でメモリに展開する（Chrome・Brave は必要時のみ個別復号）。Microsoft は「意図的な設計」と確認。管理者権限が必要なため直接的な Web 攻撃では悪用不可だが、共有環境・ターミナルサーバーでは深刻なリスク。PoC（EdgeSavedPasswordsDumperPoC）が公開済み。

**関連ページ:** [[wiki/edge-plaintext-passwords-security]]
**ソース:** [Microsoft Edge stores all passwords in plaintext RAM](https://xenospectrum.com/edge-plaintext-ram-passwords/)

### 10. 拡散モデルのフローマップ——4ステップで FLUX.1-dev をサンプリング

Sander Dieleman による拡散モデルのフローマップ（Flow Maps）解説。接線方向を逐次予測する代わりに「パス上の任意の点から任意の点を直接予測」することで劇的なサンプリング高速化を実現。合成性・ラグランジュ一貫性・オイラー一貫性の3つのルールで訓練。FLUX.1-dev を4ステップに蒸留、動画モデルを2〜8ステップに蒸留する成果。FreeFlow は元の学習データなしでフローマップを訓練できる。

**関連ページ:** [[wiki/diffusion-flow-maps]]
**ソース:** [Learning the Integral of a Diffusion Model](https://sander.ai/2026/05/06/flow-maps.html)

### 13. なぜ Rust か——C++ から移行した企業の率直なレポート

Voluntas による Rust 採用実体験レポート。採用理由3つ：①社内に Rust 経験10年超のエキスパートがいた、②C++ ビルドシステムへの疲弊、③LLM が Rust コンパイルエラーを効果的に解決してくれる（予想外のメリット）。良かった点：Cargo・Tokio・Rustls・proptest・tracing。欠点：target ディレクトリの肥大化・依存関係増大・CI キャッシュが効きにくい。
