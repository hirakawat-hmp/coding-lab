## 記事1: axiosで発見された重大な脆弱性 - 2026.04.12

Advisory ID: GHSA-fvcv-3m26-pcqx / CVE-2026-40175
CVSS スコア: 10.0（最高）

脆弱性の種類: CWE-113（CRLF Injection）とプロトタイプ汚染の組み合わせによる HTTP リクエストスマグリング

影響範囲:
- < 1.13.2: 脆弱
- 1.13.2 ~ 1.14.x: 部分修正（公式非推奨）
- ≥ 1.15.0: 安全

攻撃の仕組み:
攻撃者は依存ツリー内の脆弱性を通じて Object.prototype にCRLFを含むヘッダー値を注入。
axiosがconfig統合時にプロトタイプ由来のプロパティを取り込み、検証なくソケットに書き込むことで、
単一のHTTPリクエスト内に攻撃者制御の別リクエストを密輸できる。
最悪の場合、AWS IMDSv2認証情報を窃取しクラウドアカウント乗っ取りに至る可能性がある。

重要な注記:
記事著者は実機再現に至らず、Node.jsのhttpモジュールが「assertValidHeaderValue」により
CRLFを拒否するため、Node + axios構成では素朴なPoC再現は困難。
実際の危険性はブラウザやReact Nativeなど他の環境に限定される見込み。

対処法: npm install axios@^1.15.0

## 記事2: GHSA-fvcv-3m26-pcqx (Axiosの脆弱性) がなんか変

著者が Node.js 上でテストしたところ、`\r\n` を含むヘッダーをAxiosに渡すと「無効な文字」エラーが発生し、
ヘッダーインジェクションが機能しなかった。プロトタイプ汚染経由でも攻撃は成立しなかった。

疑問点:
- 実際にPoC（概念実証）を確認したのか
- 最新Node.jsではhttpモジュールが改行文字を検証するため脆弱性が実質化するのか
- スコア10の最高評価が適切か
- ホワイトリストではなくブラックリスト検証の必要性
