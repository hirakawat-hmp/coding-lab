## 4. プロンプティング力の評価

### 強み

- **具体的なUI構成の指示**: 「一旦ブランチを切って作業して欲しいのだが、例えば、今のプラニングタブオンリーにした上で、タブを中央で縦で2分割して2列構成にして、左側ではカードが横ではなく縦並びになるイメージ」のように、レイアウトの詳細を視覚的に伝えるプロンプトが多い。方向性だけでなく「1/4, 3/4 とかに配分する？」のような具体的な数値指定も行っている。
- **段階的な実装計画の立案**: 「まずは詳細な作業計画を立てませんか？」「ステップバイステップでやっていきたいね」「実装はTDDで行っていきたい」のように、実装前に計画を立てることを習慣的に行っている。プロジェクト全体の進め方を自ら主導している。
- **パターン提示型の意思決定**: 「パターンBが良いのかもね」「前者で良いのかも」「1でいきたいです」のように、AIが提案した選択肢から明確に意思決定を下すスタイルが確立されている。曖昧さを残さない点が効果的。
- **問題認識の先行提示**: 「A2Aで待っている間が苦痛かも」「エージェントからメッセージを取得した際に全部表示されてしまっている」のように、問題を具体的に記述してから改善を依頼するパターンが多い。
- **調査指示の明確さ**: 「特に調査して欲しいのが既存テンプレートからpresentonで使えるテンプレートの作成部分です」「バックエンドに関して詳細に調査してください。推測だが、スライド生成のロジックと、スライドテンプレート作成のロジックが含まれているんじゃないかな」のように、調査の焦点と仮説を同時に提示している。

### 改善ポイント

- **「いい感じに」の多用**: 「いい感じに実装できていると思います」「いい感じにステップを構成できないか考えていきましょうか」など、肯定的なフィードバックとして「いい感じ」が約20回使用されている。肯定の文脈が多いが、指示として使う場合は具体的な完成基準を添えるとより効果的。 → **改善提案**: 「いい感じに」の代わりに「以下の基準を満たす形で」のように、具体的な成功条件を記述する。
- **一文中に複数の論点を詰め込む傾向**: 「1. そうです。なので削除して良いものですね、新しくcloudrun を立てましょうか。2. にかんしてはその通りです。dev-sql-agent は削除で...3. に関しても、ADCは使用できないでしょうか？4. そうですね。」のように、一度のプロンプトに4-5件の指示が混在することがある。 → **改善提案**: 1プロンプトにつき1-2の論点に絞るか、箇条書きで整理して送信する。
- **タイポ・入力ミスの頻度**: 「想像つkますか？」「きちんと、書いておきましょう」「確認book根切る？」「ログ確認book根切る？」など、入力ミスが散見される。内容の理解には支障ないが、AIの解釈精度に影響する可能性がある。 → **改善提案**: 重要な技術指示のプロンプトは送信前に一度読み返す。
- **状態確認の頻度が高い**: 「今どんな感じ？」「どんな感じでしょうか」「今どう？」のような状態確認が163回出現。長時間タスクの待機中に発生する傾向がある。 → **改善提案**: 長時間タスクの場合はAIに進捗報告の粒度を事前に指定する（「各ステップ完了時に報告して」等）。

### 特徴的な癖

- **対話的な開発スタイル**: AIを同僚のように扱い、「どう思います？」「意味わかりますか？」「認識は合っていますか？」のように、常に双方向の議論を心がけている。命令ではなく協議のトーンが基調。
- **日本語ベースに英語技術用語を混ぜるコードスイッチング**: 「Agent2Agent で色々なエージェントと対話する」「Nextjs API Router でstreaming って可能なのかな？」のように、概念説明は日本語、技術用語は英語という一貫したパターン。
- **方針変更時の丁寧な前置き**: 「ごめん、先に各種の最新バージョンを確認しておきましょうか」「色々やってもらったが、これまでとは方針変更」のように、方針転換時にAIへの配慮を見せる。
- **スラッシュコマンドの活用**: `/compact`, `/mcp`, `/plugin`, `/resume`, `/context`, `/model` 等を頻繁に使用しており、Claude Code の操作体系を熟知している。

## 5. AI活用スタイル

- **アーキテクチャ設計**: エージェント構成（A2A vs 統合型）、バックエンドフレームワーク選定（Mastra vs Hono）、インフラ構成（CloudRun, CloudSQL）などの方針決定はユーザーが主導し、AIには選択肢の提示や実現可能性の検証を依頼するスタイル。「パターンBが良いのかもね」「最初から分離したいかも」のように最終判断は常にユーザーが下している。
- **UI/UXの方向性決定**: レイアウト構成、タブ構成、コンポーネント配置などの方針はユーザーが具体的に指示し、AIは実装に集中している。「左側はあくまでステッパー的な役割で、カードの内容に関しては右側のタブで」のように、方針が明確。
- **プロジェクト管理**: ブランチ戦略、コミットタイミング、PR作成、実装計画の策定など、開発プロセスの管理をユーザーが主導している。「まずは詳細な作業計画を立てませんか？」「TDDで行っていきたい」のように開発方法論も指定。

### AIへの依存が高い領域

- **OSSの調査・技術リサーチ**: 「presenton」「screenshot-to-code」「mastra」などの調査をほぼ全面的にAIに委任している。「まずは調査しましょうか」「バックグラウンドで調査を走らせましょうか」のパターンが頻出。 → **自立に向けた提案**: 調査結果をAIに要約させた後、公式ドキュメントを自ら読んで理解を深めることで、次回以降より具体的な指示が出せるようになると推測されます。
- **デバッグ・エラー解決**: 「エラーが出ました」「何も表示されていないです」「うんともすんとも言わなくなった」のように、症状を伝えてAIに原因究明から修正まで一括で依頼するケースが見られる。 → **自立に向けた提案**: エラーログの事前解析や、問題の切り分け（フロント/バック/インフラ）を自ら行ってからAIに相談すると、解決速度が向上すると推測されます。
- **ドキュメント・コード生成**: 「docs/配下にまとめて」「ここまでの知見をまとめて」「トラブルシューティングに書いておきましょう」のように、ドキュメント作成をAIに全面委任する傾向がある。 → **自立に向けた提案**: 重要な設計判断や学びはユーザー自身の言葉で記録する習慣をつけると、知識の定着につながると推測されます。

### ツール別の活用傾向

## 6. 成長の軌跡と学習提案

### 観察された成長

- **エージェント構成の理解深化**: 2月上旬は「チャットエージェントだけで良いのかな」と模索していたのが、2月下旬〜3月にかけて「Agent2Agent 構成」「Mastra AgentNetwork」「ADK + A2Aクライアント」と、より具体的なアーキテクチャパターンを自ら提案するようになっている。
- **インフラ知識の実践的獲得**: 2月24日のGCPインフラ再構築セッションを経て、CloudRun / CloudSQL / Cloud Armor / Secret Manager / IAM の関係を実践レベルで理解し、以降は「Invokerの設定」「Authorized Network」「ADC (Application Default Credentials)」などの用語を正確に使いこなしている。
- **新技術領域への挑戦**: 2月の atl-document-generator を主軸としつつ、3月に入って minimal-groq-server（音声対話）、audio-reactive-mcmc（科学計算＋ビジュアライゼーション）、agent-development（Google ADK）と、異なる技術ドメインへの探索が広がっている。

### 繰り返し発生する課題

- **コンテキスト切れへの対応**: `/compact` や `/resume` の頻繁な使用（15回以上）から、長時間セッションでコンテキスト上限に達するケースが多い。特に atl-document-generator の複雑なセッションで顕著。
- **方針変更の繰り返し**: 「色々やってもらったが、これまでとは方針変更」「ごめん、先に...」のように、実装途中での方針転換が複数回発生している。事前の設計フェーズでより詳細な検討を行うことで軽減できる可能性がある。
- **環境設定の再構築**: Docker リビルド、パッケージ更新、環境変数の再設定が繰り返し発生しており、開発環境の再現性に課題がある可能性がある。

### 推奨される次のステップ

1. **コンテキスト管理戦略の確立**: 長時間セッションでの `/compact` 頻度を減らすため、(a) セッション開始時に AGENTS.md やプロジェクトメモリに最新の状況を書き出す、(b) 1セッション = 1テーマに絞る、(c) セッション冒頭で「前回のセッションでXを完了、今回はYに着手」のようなコンテキスト要約を渡す、といった運用を検討する。
2. **デバッグスキルの強化**: エラー発生時に「エラーログの全文を貼り付ける前に、自分で原因の仮説を立ててからAIに相談する」というワークフローを試すと、問題解決の自立度が向上する。特に音声処理（minimal-groq-server）のような不慣れな領域では、基礎知識の学習と並行してAIを活用すると効果的。
3. **設計ドキュメントの充実化**: atl-document-generator のようなフラッグシップ・プロジェクトでは、アーキテクチャ決定の理由（ADR: Architecture Decision Records）を記録する習慣を持つと、方針変更時の影響範囲の把握や、新しいセッションでのコンテキスト共有が容易になる。
